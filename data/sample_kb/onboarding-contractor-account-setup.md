# SOP: Temporary or Contractor Account Setup

## Issue
A contractor, temp worker, or intern needs system access for a fixed-term engagement, with access scoped and time-limited appropriately.

## Resolution Steps
1. Confirm the engagement details from the hiring manager: exact start and end dates, systems needed, and whether this is an internal-style account or external guest access (see SOP "Guest / External User Access Requests" for external partners specifically).
2. For contractors needing a full internal-style account, create it with a clear naming convention distinguishing it as non-employee (e.g. a department or vendor tag) so it's identifiable in audits later.
3. Scope access tightly to only what the engagement requires — contractor accounts should never inherit a broad standard-employee access template by default.
4. Set an account expiration date matching the contract end date wherever the identity platform supports it — this prevents the single most common audit finding: stale contractor accounts still active months after engagement end.
5. Skip provisioning any standing distribution list memberships or company-wide communications access unless specifically required for the role.

## Common Follow-up Issues
- Contract gets extended — this needs a new ticket to formally extend the expiration date, not an assumption that access continues automatically.
- Contractor needs a company device — evaluate against policy; many contractor engagements use BYOD with restricted access instead.

## Escalation
Contractor access to sensitive systems or data requires manager and, depending on sensitivity, Legal/Security sign-off before provisioning — flag and hold rather than provisioning by default.

## SLA
P3 (time-sensitive engagement start) — target resolution 1 business day.
