A plain-language walkthrough of how a question typed into the chat becomes a grounded, source-cited answer — no ML background required to follow it.

The pipeline

01
You ask a question
Plain English, typed or tapped from an example — e.g. "my vpn keeps disconnecting."
↓
02
Embedding
Your question is converted into a list of numbers (a "vector") that captures its meaning — done locally with a small ONNX model, not sent anywhere yet.
↓
03
Retrieval
That vector is compared against every pre-embedded knowledge-base article to find the closest matches — this is ChromaDB doing similarity search across 50 SOPs in milliseconds.
↓
04
Context building
The top 3 matching article excerpts are pulled together into a single block of text — this becomes the only source of truth the AI is allowed to answer from.
↓
05
Generation
The question + that context are sent to an LLM (Llama 3.1 via Groq) with strict instructions: answer only from what was retrieved, translate technical steps into plain language, and say "I don't know" rather than guess.
↓
06
Streaming response
The answer streams back word by word as it's generated, rather than making you wait for the full response — plus the source article(s) it came from, a confidence score, and (when relevant) a real action button, like the self-service password reset.
Why this matters: this is the same core pattern (retrieval-augmented generation, "RAG") used in production tools like enterprise Copilot search and internal AI help bots. The point of grounding answers in retrieved documents instead of asking the model to answer from memory is that it can't invent a fix that doesn't exist — it can only work from the real, approved SOPs it was given.

What's actually running

LayerWhat it does
FrontendPlain HTML/CSS/JS — no framework overhead for a tool this size
BackendPython, FastAPI, streaming responses over Server-Sent Events
EmbeddingsChromaDB's built-in ONNX MiniLM model — chosen specifically to fit free-tier hosting memory limits (see the deployment notes for what happens if you don't make this choice)
Vector storeChromaDB, running embedded — no separate database server needed
LLMLlama 3.1 8B via Groq's API — fast inference, generous free tier
HostingRender, free tier — rate-limited to protect API quota

Deliberate design decisions

A few choices here were made on purpose, not by default — worth knowing if asked about them:

No PyTorch. The obvious choice for embeddings is a sentence-transformers model, but that pulls in PyTorch — which alone can exceed a free hosting tier's memory limit. Swapping to ChromaDB's lightweight ONNX embedding function does the same job in a fraction of the memory footprint.
No chatbot password resets. An unauthenticated chat interface should never be trusted to reset a password or unlock an account — that's an identity-verification problem, not a chat problem. Password-related questions instead hand off directly to Microsoft's own SSPR portal, which does real MFA-based verification.
Confidence isn't decorative. The confidence score shown on each answer is computed from the actual vector distance between your question and the best-matching article — not a made-up number. Low confidence means the retrieval genuinely didn't find a strong match, which is a signal to raise a ticket rather than trust the answer.
